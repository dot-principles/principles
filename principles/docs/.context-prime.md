# .principles prime context — docs
# Sections: Principle · Why it matters · Good practice

### DOC-PURPOSE — Every document has one clear purpose

A document should serve exactly one reader goal. The four fundamental documentation types — tutorials, how-to guides, reference material, and explanations — serve different purposes and should not be mixed.

Why it matters:

Mixed-purpose documents disorient readers because each documentation type requires a different reading posture. A reader following tutorial steps does not want to be interrupted by reference tables. Documents that serve multiple purposes satisfy none well, grow without bound, and become hard to maintain.

Good practice:

- Choose one documentation type before you start writing: tutorial, how-to, reference, or explanation
- State the document's purpose in the first sentence or opening section
- Link to related documents of other types rather than embedding their content
- Use the Diátaxis framework as a guide: each type has a distinct structure and tone

### DOC-MINIMAL — Write only what is necessary

Apply YAGNI to prose: do not document speculative features, rarely-used edge cases, or information the reader can derive from context. Every sentence has a maintenance cost.

Why it matters:

Over-documented systems create an illusion of completeness while hiding what matters under layers of noise. Readers learn to skim and miss critical information. Maintainers face an ever-growing surface area to keep accurate.

Good practice:

- Start with the minimum viable document: the one thing the reader must know to succeed
- Delete content that was true once but is no longer relevant
- Challenge every paragraph: what does the reader do differently after reading this?
- Apply the same code-review discipline to documentation PRs that you apply to code PRs

### DOC-AUDIENCE — Write for a specific, named audience

Every document should be written for a specific, identifiable reader with a known level of existing knowledge and a concrete goal.

Why it matters:

When a document lacks a defined audience, authors hedge — they over-explain concepts experts find patronizing, while glossing over assumptions that beginners need spelled out. The result is a document that neither audience finds useful.

Good practice:

- State the target audience and prerequisite knowledge in the first paragraph or "Before you begin" section
- If multiple audiences need the same topic, write separate documents or clearly labeled sections
- Use the audience's vocabulary, not internal jargon, unless you define it on first use
- Test the document with a representative reader from the intended audience before publishing

### DOC-ACCURACY — Keep documentation accurate and synchronized with the system

Documentation must accurately describe the current state of the system it documents.

Why it matters:

Inaccurate documentation causes wasted debugging time, incorrect integrations, and eroded trust. Once readers learn that docs are unreliable, they stop consulting them and resort to reading source code or asking colleagues.

Good practice:

- Include documentation updates in the definition of done for every code change that affects observable behavior
- Run documentation examples as part of the CI pipeline where possible (doc-tests, code samples in tests)
- Review documentation accuracy during API reviews and architecture reviews
- Mark known-stale sections explicitly rather than leaving readers to discover inaccuracies at runtime

### DOC-EXAMPLES — Illustrate with concrete, runnable examples

Abstract descriptions should be accompanied by concrete examples a reader can try immediately.

Why it matters:

Abstract descriptions leave readers uncertain whether they have understood correctly. A reader who must mentally simulate a system from prose alone is doing extra work that a good example would eliminate.

Good practice:

- Lead with the simplest possible working example before presenting variations or options
- Use realistic values in examples (real-looking URLs, plausible field names, actual command output)
- Show both input and output for any transformation or command
- Keep examples short: strip everything that does not contribute to illustrating the principle
- Test examples in CI where possible

### DOC-PROGRESSIVE-DISCLOSURE — Reveal complexity progressively

Present information in layers of increasing complexity: begin with the simplest case, then progressively reveal advanced options, edge cases, and deeper explanations.

Why it matters:

Front-loading complexity drives away readers who only need the simple case. Hiding advanced detail at the end frustrates experts who need it quickly.

Good practice:

- Open with the simplest possible working version of the thing being documented
- Structure headings to signal depth: "Basic usage" → "Configuration" → "Advanced options" → "Troubleshooting"
- Put the most common use case first; treat edge cases as separate sections
- Write a one-paragraph summary or TL;DR at the top of long documents
